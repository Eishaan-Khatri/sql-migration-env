System Architecture

How It Works Under the Hood

A modular, three-tier architecture combining an isolated SQLite sandbox, a dynamic golden-database grader, and a standards-compliant OpenEnv API server.

🧬

Dynamic Golden Grader

Instead of hardcoding expected outputs, the grader programmatically regenerates the perfect database state from golden migration functions in seeds.py. Change the seed data → the grader automatically recalibrates. Zero manual maintenance.

📊

Live ERD Visualization

Every observation includes a Mermaid.js erDiagram dynamically parsed from sqlite_master and PRAGMA foreign_key_list. The agent can "see" the current table relationships at each step, enabling spatial reasoning about schema drift.

⚡

Efficiency Metrics

Evaluates not just correctness but computational cost. Integrates SQLite's sqlite3_progress_handler to track VM instruction count (ops) and wall-clock execution_ms per query. Agents are graded on performance, not just outcomes.

📜

Pydantic Data Contract

Formal API contract via MigrationAction, MigrationObservation, and MigrationState — all extending OpenEnv base types. Guarantees type safety, validation, and schema-level interoperability with any RL training framework.

Scoring Engine

The Reward Function

A multi-dimensional, weighted composite score designed to prevent token-hacking and encourage genuine schema understanding.

Rt = Pt − Pt−1

Where Pt (Migration Progress) is a weighted composite bounded to [0.01, 0.99]:

30%

Schema Match

Tables exist with exact (name, type, notnull, default) column signatures.

40%

Data Integrity

Row content mapped against Golden DB with 10⁻⁷ float epsilon tolerance.

20%

FK & Constraints

Foreign keys enforced, PRAGMA integrity_check passes, no orphans.

10%

Anti-Exploit

Penalty for schema pollution, empty tables, or row bloating attacks.

Adversarial Hardening

Security & Anti-Exploit System

A multi-layered defense preventing agents from gaming the benchmark or escaping the sandbox.

🚫 SQL Blacklist Regex

Prevents agents from escaping the sandbox via filesystem-level database access or dangerous PRAGMA modifications.

_DANGEROUS_PATTERNS = re.compile(
    r"\b(ATTACH\s+DATABASE|DETACH\s+DATABASE"
    r"|LOAD_EXTENSION)\b"
    r"|PRAGMA\s+writable_schema",
    re.IGNORECASE,
)

⏱ VM Instruction Counter

SQLite's progress_handler terminates queries exceeding 500,000 VM operations. Prevents infinite CTEs and recursive attacks.

_MAX_OPS = 500_000

def _progress_callback():
    ops_count[0] += 1
    if ops_count[0] > _MAX_OPS:
        return 1  # Non-zero = abort
    return 0

conn.set_progress_handler(
    _progress_callback, 1000
)

🔒 PRAGMA State Preservation

The grader carefully saves and restores the agent's foreign_keys PRAGMA state before scoring, preventing the grading engine itself from corrupting the database being evaluated. This guarantees scoring isolation.

🧪 Row Bloat Detection

Applies a 1% penalty per extra row above the golden expected count. Prevents agents from "guessing" by dumping all possible data combinations. Empty table exploitation is also detected and penalized.

Contributing a New Task (3 Steps)

The benchmark is designed to be extended. Adding a new adversarial task requires only three additions to seeds.py:

# Step 1: Define seed data and the seed function
TASK9_SOURCE_DDL = """CREATE TABLE ..."""
TASK9_SOURCE_DATA = [(1, "..."), ...]

def seed_task9(conn):
    conn.executescript(TASK9_SOURCE_DDL)
    conn.executemany("INSERT INTO ...", TASK9_SOURCE_DATA)
    conn.commit()

# Step 2: Define the golden (correct) migration
def golden_task9(conn):
    conn.execute("CREATE TABLE new_table ...")
    conn.execute("INSERT INTO new_table SELECT ... FROM old_table")
    conn.execute("DROP TABLE old_table")
    conn.commit()

# Step 3: Register in the TASKS dictionary
TASKS["my-new-task"] = {
    "seed_fn": seed_task9,
    "golden_fn": golden_task9,
    "target_ddl": "CREATE TABLE ...",
    "description": "A description of the task.",
    "difficulty": "hard",
    "max_steps": 20,
}
